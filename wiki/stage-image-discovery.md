# Image Discovery

> [!info] Pipeline role
> Build a broad cross-institution **thumbnail board** of an artist's works for human curation, then resolve only the SELECTED works to the best legally-clear high-resolution image. Emits the curation board, then `images/selected/` + per-image metadata.

## What the research says

[[16.1-image-source-hierarchy]] establishes a tiered source hierarchy. **T1 — Wikidata** is the pivot/identity layer: a `creator (P170)` SPARQL query returns an artist's works across every holding institution, with the QID as the universal key for cross-source dedup (institution-id properties P4610 AIC, P3634 Met bridge to museum records). **T2 — Met / AIC / Cleveland** provide the highest-resolution assets and machine-readable rights (`isPublicDomain`, `is_public_domain`, `share_license_status=="CC0"`). **T3 — Wikimedia Commons (SDC) / Europeana** are broad discovery aggregators. **T4 — DPLA / Google Arts / WikiArt / general image search** are discovery-only (no reliable rights) and must be verified through T1/T2 before any high-res download.

The IIIF acquisition mechanics live in [[concept-iiif]] (from [[03.1-museum-image-apis]]): the `info.json`-then-`max` fetch pattern and automated rights parsing. Commons thumbnails are reached via `Special:FilePath/<file>?width=N`.

## Open questions / tensions

- **Namesake/foundation QID collisions** — name search returns both the artist and lookalike entities (e.g. "Zentrum Paul Klee"). Disambiguate by work-count (the real creator has the works); surface to the user on a genuine tie.
- **Coverage vs rights** — the board is intentionally rights-agnostic (browse everything); rights only gate the post-curation high-res download. Missing rights ⇒ treat as restricted, keep the thumbnail/source link only.
- **Recall vs precision on attribution** — Wikidata `creator` discovery is high-recall and authoritative; the legacy Commons keyword guard was precision-first (could drop correct files). Wikidata supersedes it for discovery; the keyword path survives only inside the resolver.
- **WDQS is a single point of failure (F1).** A live Klee run hit a WDQS outage: the heavy `qid_lookup_sparql` (label-filter + work-count aggregation) `504`'d while a trivial label query still `200`'d — the headline Wikidata board was unusable for the whole session ([[18-uat-feedback]]). Resolve QID via the lighter entity-search API (`wbsearchentities`) and run the work-count tiebreak only on genuine ambiguity; make AIC-only a *designed* degraded fallback (clear board banner, stage marked `degraded` in state) rather than a crash.
- **Posthumous / reproduction attribution (F3).** The surname/agent-id match let a 1972 Montgomery Ward textile "after a painting by Paul Klee" onto the board. Guard against `after | produced by | manner of | copy after | workshop of` in `artist_display`, a `date_display` year past death + margin, or a reproduction classification — flag at minimum.
- **Zero-PD artists (F5).** For artists like Klee (d. 1940) the board can be 100% in-copyright with nothing for the high-res phase to resolve; surface a board notice so the human expects analysis to lean on graded sources + reference links, not redistributed images.
- **Stateful, re-enterable discovery — BUILT (Thrust 1).** [[19-stateful-runs-custom-images-staged-analysis]] Thrust 1 is implemented (spec `docs/superpowers/specs/2026-06-20-stateful-package-state-design.md`, merged 2026-06-20): `state.json` is now one fat document `{artist, completed, runs[], candidates[], sessions[]}`. `PackageState.merge_candidates` folds a discovery board into the accumulating `candidates[]` idempotently — QID primary → `inst_ids` → `work_id` key, never duplicating or clobbering the original `first_run`; `record_run` ledgers each run and carries the `degraded` flag, so a later run can **upgrade** an AIC-only fallback once WDQS recovers. The merge key resolves the open state-format/audit questions. *Still open:* the F1 lighter-QID/`degraded`-path and F3 reproduction guard above are independent of state and remain to build.
- **User-supplied images as a first-class source — BUILT (Thrust 2).** [[19-stateful-runs-custom-images-staged-analysis]] Thrust 2 is implemented (spec `docs/superpowers/specs/2026-06-20-custom-image-injection-design.md`, merged 2026-06-20): a re-enterable **import** operation folds the user's own image folder into `candidates[]` as `origin:"user"`. Identification chose **Claude vision → pipeline verify** over an external reverse-image API: Claude (the agent) proposes `{artist, title, date}` per image and the *existing* discovery pipeline (`make_pipeline_lookup` over `search_wikidata`/`search_aic`) corroborates it — `verify_identification` lands each as `confirmed` (pipeline hit → authoritative `qid/museum/source_url/rights`), `proposed` (guess, no record → `rights:"unknown"`), `off_artist` (set aside — one folder = one artist), or `unidentified`. A **batch trust gate** (`build_review` → `import-review.json/.html`, human edits/confirms, `parse_review`) precedes the board (mirrors the F3 "don't trust auto-derived provenance" lesson). `ingest_import_review` copies files into `images/user/` and merges via `merge_user_candidate`: a new work appends as `origin:"user"`; one already on the board (QID/`inst_ids`/`work_id` dedup) is **enriched** with the local file (`local_path`) rather than duplicated. An unverified (no-QID) image is identified by its source file, so re-import is idempotent. Visual analysis reads `local_path` directly — no rights-gated re-download. Thrust 3 (the narrowing funnel + persistent-star/select gallery) has since shipped too, so **raw/19 is now fully built** — see [[stage-curation]].
- **Image collection is the weakest link (M2 UAT, live Monet run — point of concern, MONITOR).** Across runs the T1/T3 layers (Wikidata + Wikimedia Commons) have been usable only for brief windows; the Monet run again **lost WDQS entirely** (3 ReadTimeouts → designed AIC-only degraded board, 46 PD works). Board breadth/recall is the shakiest part of the whole pipeline and the user flags it as the top concern to watch going forward. Candidate directions floated (none decided): an **exhaustive / longer collection pass**, possibly **asynchronous** — kick off a background collection that accrues candidates over time and folds them into `candidates[]` via the built stateful `merge_candidates`/`record_run` upgrade path (the storage already supports incremental accrual; the open question is the *collection strategy*, not the schema) — and/or leaning harder on the T2 museum APIs (Met/AIC/Cleveland) when T1/T3 are down. Overlaps the still-open F1 lighter-QID/`degraded` work above. **Candidate-source backlog** (running list in `TODO.md` → "Backlog — candidate image sources"): user-suggested **Artvee** (artvee.com) — curated high-res public-domain artwork, free + paid membership for max-res (the user subscribes); no API (scrape via Firecrawl/Scrapy), PD-curated but verify, dedup by artist+title, treat as a user-credentialed **T3 supplement/fallback**.
- **`works.md` canon absent from the board (M3 UAT, live Monet run — user-classified FAILURE).** The Monet `works.md` (Claude-authored inventory) listed the famous, learnable works — Haystacks, Rouen Cathedral, Gare Saint-Lazare, Water Lilies — but **many never appeared on the gallery board**, so the human could neither curate nor learn from them. Causes compound: WDQS down → AIC-only board (every non-AIC canonical work vanishes), **plus** no reconciliation between the canon Claude writes in `works.md` and what discovery actually surfaces — the inventory promises works the board can't deliver. Needs: (a) a **`works.md` ↔ board reconciliation** that flags every inventory entry with no candidate and actively searches for it (per-work targeted lookup, not just the bulk creator query); and (b) the reduced single-source fragility of M2. The most serious finding of the run.
- **Perceptual duplicate detection for the deferred re-query dedup spec — research in hand ([[22.1-image-duplicate-detection]]).** The built `merge_candidates` dedup key is *identifier-based* (QID → `inst_ids` → `work_id`); it can't catch the same artwork arriving from two sources with no shared id (the deferred duplicate-handling-on-re-query item in `TODO.md`). [[22.1-image-duplicate-detection]] surveys the image-similarity fallback: normalize any method to a 0–1 score (`S = 1 − D/64` for a Hamming-distance hash; `S = (cos+1)/2` for embeddings) with a tunable threshold (~0.85 = "same artwork"; ≥0.95 strict). The report's headline pick is **CLIP/CNN embeddings via `imagededup`**, optimized for *rotation/viewpoint* robustness. **Tension to resolve at spec time:** our case is the same painting across museum sources (resolution/crop/JPEG/color-cast/watermark, **not rotated**) and identifier-less only in the fallback — so the far lighter pure-Python **`imagehash` pHash+wHash** likely suffices as primary, reserving CLIP for genuinely reframed/identifier-less cases. Favor a **hash pre-filter → embedding-confirm** hybrid only if false positives demand it. **Engine BUILT (Spec A, 2026-06-21):** chose pure-Python **`imagehash` pHash+wHash** (CLIP rejected — over-collapses crops; "only same full image" is the rule), conservative `score = min(pHash_sim, wHash_sim)`, `DUP_THRESHOLD = 0.90`. Three offline modules merged to main (8-task→4-task subagent TDD, 293 tests): `scripts/image_similarity` (0–1 score, fail-open), `scripts/image_manifest` (`manifest.json` = provenance record **and** dedup index via stored-hash `find_match`), `scripts/dedup` (pure keep/merge/add decision: largest-pixel-area wins, metadata fills empty fields, **stars preserved**, Obsidian-safe canonical rename — no filesystem I/O). Spec `docs/superpowers/specs/2026-06-21-image-dedup-engine-design.md`; plan `docs/superpowers/plans/2026-06-21-image-dedup-engine.md`. **Consumer BUILT (Spec B, 2026-06-21) — feature complete.** Library-collection mode merged to main (8-task subagent TDD, 314 tests, **live Cezanne e2e passed**): new `scripts/library.py` (`execute_action` atomic place→delete, `build_library` batch dedup, `make_incoming`, `sync_candidates` manifest→board with **stars preserved on re-sync**, `seed_import` external→`images/library/` with `images/user/` emptied — externals never touched), `image_download.download_library` + `resolve.default_resolve_url` (eager full-res → `images/incoming/`), and a SKILL Phase-A library mode that **coexists** with the thumbnail-only board and feeds the *existing* funnel unchanged. Manifest (`images/manifest.json`) is both provenance ledger and dedup index; downloads rights-recorded-not-gated (user-authorized). Live run: Wikidata degraded→AIC-only, **30 Cezanne seed files → 26 deduped** (byte-identical pair + Madame Cézanne JPG↔60MB PSD collapsed), 20 downloaded/16 net-new, 42 final entries. Spec `docs/superpowers/specs/2026-06-21-library-collection-mode-design.md`; plan `.../plans/2026-06-21-library-collection-mode.md`. Final-review fix: `museum_search.display_url` now honors `local_path` for any origin (was user-only) so discovered library cards zoom correctly. **Carry-forward:** metadata merge is existing-non-empty-wins, so a seed-first filename title persists even after QID-rich discovery — the deferred **QID-authoritative-override** is the only remaining dedup follow-on (out of scope here).
- **User-image injection as a canon-gap mitigation (M4, planned next UAT).** The next UAT will deliberately drop **user-provided images** into the study (the BUILT Thrust-2 import path above) to backfill the famous works discovery missed — testing whether custom images are a practical workaround for M2/M3. Worth watching whether import becomes a *routine* step (not an exception) whenever discovery under-delivers, which would itself argue for fixing M2/M3 upstream.

## Skill design implications

Two phases:
1. **Discovery → board (rights-agnostic).** Wikidata is the primary source (`search_wikidata`): resolve the artist QID, query works, build thumbnail candidates (Commons `Special:FilePath`). AIC supplements works Wikidata lacks; `merge_boards` dedups by QID → AIC/Met inst-id → folded title+date. The board shows many thumbnails for the human to rate.
2. **Resolution → selected (rights-gated).** For each selected work, a pluggable resolver chain fetches the best legally-clear high-res: Commons P18 (PD/CC0) → AIC IIIF 1686px (`is_public_domain`) → else keep `source_url` (in-copyright, thumbnail only). Met/Cleveland resolvers drop into the same interface later.

Copyright posture: browse thumbnails ≤843px (hotlinked, for private curation); download high-res ONLY on a verified PD/CC0 flag. Capture per-image metadata: source URL, institution, license, pixel dimensions, QID, and parent work id. Never auto-select — curation is human ([[stage-curation]]).

Discovery now writes into persisted package state (Thrust 1, built): the merged board is stored as `candidates[]` (each entry tagged `origin: discovered|user` — the `user` value is now written by the built Thrust 2 import path; user cards carry a `local_path` and show a `USER` badge on the board) and every run is appended to `runs[]`. Re-running discovery merges rather than replaces, so a later run after a WDQS outage extends the same board.
