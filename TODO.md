## Backlog — feature request: Apple Photos ingest channel (Facebook-group-sourced curation)
**Requested 2026-06-21.** A new way to get the user's *own already-curated* images into a study. The user follows **artist-specific Facebook groups** (group name ≈ artist name), "likes"/saves images they like, and those land in **Apple Photos**. The like *is the curation gesture* — it happens upstream, on Facebook. The ask: let the skill harvest those saved images as a seed source, identify them, recover where they came from, and mine the user's likes for **stats that point back to high-yield feeds**. This is an **ingest channel**, not a museum-API source — it feeds the *existing* identify + dedup pipelines, so most of the machinery already exists. Decompose into integration levels (build L1 first; each is independently useful):

- [ ] **L1 — Apple Photos ingest adapter (MVP).** Read images from a designated Photos album / smart album (convention: one album per artist/group), export the originals to a temp dir, then route them through the **already-built** path: Thrust-2 custom-image injection (`scripts/user_import.py`: Claude-vision → pipeline-verify → batch trust gate) for identification, then **library/dedup mode** (`scripts/library.py` `seed_import`/`build_library`) for folding + quality-winner + provenance manifest. Net new code is mostly the *adapter* (Photos → temp folder), not the pipeline. Likely lib: **`osxphotos`** (mature Python reader for the macOS Photos library — albums, keywords, persons, captions, EXIF, original export); macOS-only, read-only. **Open:** album-convention vs smart-album-by-keyword for selecting "the FB-saved set"; how to mark already-ingested photos so re-runs are idempotent (Photos UUID → run ledger).
- [ ] **L2 — provenance / source-feed recovery.** Want per-image `{source_feed (which FB group), artist, year}`. **Tension:** Facebook strips embedded EXIF/IPTC on save, so the feed name is usually *not* in the file. Recoverable instead from **how the user files it in Photos** — album name (= group/artist), keywords/tags, or a caption the user adds — i.e. a **user convention**, not embedded data. Define the minimal convention (e.g. "album name = FB group name"); fall back to Claude-vision identification (already in L1) for artist/title/year when filing is sparse. Record source as an `origins[]` entry (`source:"apple-photos"`, `source_feed:"<group>"`, `rights:"unknown"`) on the manifest — matches the library mode's recorded-not-gated posture for a private study.
- [ ] **L3 — like-stats + feed exploration.** Aggregate the user's likes into analytics: **how many liked images came from each feed/artist**, rank feeds by yield, surface "this feed is worth exploring for more." Two sub-levels: (a) **stats only** over what's already in Photos (cheap, just a group-by on the L2 `source_feed`); (b) **actually crawl the FB group feed** for more images — **HARD / gated:** Facebook group content is auth-walled and scraping it likely violates FB ToS; treat as a separate, explicitly-flagged investigation (API access is effectively closed for this use; a user-credentialed/manual-export path may be the only viable route). Don't fold (b) into L1/L2.

**Why it fits:** slots in *front of* the existing custom-image-injection + library/dedup pipelines (so identification, quality-winner, stars-preserved, and the provenance manifest all come for free) and directly attacks the **M2/M3 image-collection-breadth** weakness — the user's own curated likes are a high-precision supplement when Wikidata/Commons/museum discovery under-delivers. Related: [[stage-image-discovery]] open questions (M2/M3), the BUILT Thrust-2 import + library mode, and the **Artvee** entry above (another user-credentialed source). When promoted to active work: brainstorm → spec per the superpowers flow; **L1 is the buildable MVP.**

## Backlog — feature request: works.md ↔ gallery reconciliation + targeted image backfill
**Requested 2026-06-21.** Promotes the **M3 finding** ("`works.md` canon absent from the board" — *the most serious finding* of the Monet run, see [[stage-image-discovery]]) from an open question to an explicit feature. **The gap:** `works.md` is the Claude-authored inventory of the artist's primary/most-famous works, but the gallery board is built from *discovery* (Wikidata creator query + AIC) — so canonical works routinely **never reach the board** (acutely when WDQS is down → AIC-only), and the human can't curate or study the very works the inventory promised. **The ask:** a pass that guarantees every primary work listed in `works.md` is present in the image gallery; for any that are missing, do **backup web searches / "ferret around the internet"** to find images. Decompose (build L1 first; each independently useful):

- [ ] **L1 — reconciliation / gap report.** After the board is built (or `works.md` written), match each `works.md` entry against `state.candidates` and emit the **gap list**: canonical works with **no** board candidate. Even surfacing the gaps (a board/report notice + a `works.md` "on board ✓ / missing ✗" column) is high-value on its own. **Hard part = matching:** `works.md` titles are free-text (Claude-authored) vs board candidates carrying museum titles / Wikidata labels / `inst_ids` — needs **fuzzy title (+ year) matching**, and must handle **one-inventory-entry → many-board-candidates** series (e.g. *Mont Sainte-Victoire*, the 5 *Card Players*). No silent drops.
- [ ] **L2 — targeted structured backfill.** For each gap, run a **per-work targeted lookup** (not the bulk creator query): title search against **Wikidata/Commons** and the **museum APIs** (Met/AIC/Cleveland). Fold any hit through the **existing dedup/library pipeline** (`make_incoming`→`build_library`→`sync_candidates`) so it lands on the gallery with provenance + rights. This is the direct fix for the WDQS-down / bulk-query-miss cause.
- [ ] **L3 — web-search "ferreting" backfill.** When the structured APIs can't supply a canonical work, fall back to **general web image search** (WebSearch + WebFetch/Firecrawl, or forward image search). **Two tensions to resolve at spec:** (a) **trust** — arbitrary web images are unreliable, so each candidate must pass a **Claude-vision verify** that it actually depicts the named work (reuse Thrust-2 `verify_identification`); (b) **rights** — web-ferreted images carry **no rights metadata**, so either record-not-gated (`rights:"unknown"`, private-study posture, like library mode) or keep as **reference-link-only**. Works that *still* can't be found get a **"could not locate" notice** to the user — never silently dropped.

**Why it fits:** L2/L3 reuse the BUILT dedup/library fold + Thrust-2 verify; only the **reconciliation/matching** (L1) and the **web-search adapter** (L3) are net-new. Strongly coupled to the M2 breadth work (the F1 lighter-QID/`degraded` path) — together they're the answer to the canon-gap. Related: [[stage-image-discovery]] M3 (a)+(b), the Apple-Photos and Artvee user-credentialed sources above. When promoted: brainstorm → spec; **L1 (gap report) is the buildable MVP** and de-risks the matching problem before any fetching is built.
