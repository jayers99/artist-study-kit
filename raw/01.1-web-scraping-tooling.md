# Comparative Analysis: Web Extraction Stacks for Art-Historical Research

## 1. Workload Requirements & Thematic Analysis

As museum and institutional archives transition from static repositories to dynamic, Single-Page Applications (SPAs), the technical requirements for data extraction have fundamentally shifted. In my role as a technical architect, I categorize our art-historical research requirements into three distinct engineering jobs:

1.  **Clean Article Extraction (Museum Essays):** High-fidelity retrieval of scholarly text from museum portals. This necessitates the removal of aggressive "boilerplate" (navigation menus, cookie banners, and sidebar widgets) to produce clean Markdown. Markdown has become the non-negotiable standard for research because it is token-efficient for LLMs and preserves semantic document structure for Retrieval-Augmented Generation (RAG) pipelines.
2.  **Discovery on JS-heavy Institutional Sites:** Modern institutional sites often return empty HTML shells, relying on client-side JavaScript to populate exhibition records. Discovery requires tools that can render JS and map complex site structures without the overhead of full-page scrapes.
3.  **High-Resolution Image/Metadata Procurement:** This involves the concurrent capture of high-resolution media and structured metadata (provenance, dates, artist attributions). This job requires robust storage orchestration (S3/GCS) and media processing capabilities.

"Static" scraping is no longer sufficient; it fails on modern frameworks and lacks the transformation logic required for AI-ready data. We require a stack that balances deep interaction with high-performance heuristics.

## 3. Specialized Tool Profiles

### 3.1 Firecrawl: The Managed Multi-Tool
Firecrawl is the leading API-first solution for developers who prioritize scale over infrastructure management.
*   **Performance & Coverage:** It delivers a 50ms average response time with 96% web coverage, significantly outperforming traditional cURL-based scrapers.
*   **Agentic Capabilities:** It features a dedicated **Autonomous Research Agent (`firecrawl_agent`)** that follows links and synthesizes findings independently. It also includes the **Model Context Protocol (MCP) server**, allowing AI agents like Claude to drive crawls directly.
*   **Discovery & Interaction:** The `/map` endpoint provides sub-second URL discovery across entire domains. For content behind complex forms, the `/interact` endpoint allows for natural language browser automation.

### 3.2 Crawl4AI: The Local-First Python Powerhouse
Crawl4AI is an asynchronous toolkit designed specifically for local RAG pipelines where data sovereignty is paramount.
*   **Adaptive Learning:** It employs "Adaptive Web Crawling" which **learns reliable CSS selectors over time**, reducing the maintenance burden as museum site layouts drift.
*   **Resource Management:** Its **Memory-Adaptive Dispatcher** scales workers based on available system RAM, while its LXML-based parsing strategy offers up to 20x speed improvements over standard parsers.
*   **AI Filtering:** Includes BM25 content filtering to rank and keep only sections relevant to specific research queries.

### 3.3 Scrapy: The Industrial Crawler
For multi-million page crawls, Scrapy remains the gold standard for state management and media orchestration.
*   **Industrial Media Pipeline:** Scrapy's built-in image processing handles SHA1 hashing for filenames, thumbnail generation, and minimum size filtering.
*   **Storage Integration:** It natively supports uploading high-res archives to **Amazon S3 or Google Cloud Storage (GCS)**, which is critical for long-term art-historical image preservation.
*   **Orchestration:** While it requires the `scrapy-playwright` plugin for JS rendering, its middleware system handles complex retries and proxy rotation better than any library.

### 3.4 Playwright: The Automation Foundation
A browser automation toolkit that offers maximum control at the cost of significant resources.
*   **Interaction Control:** Essential for museum sites with complex lazy-loading or content hidden behind user-triggered events.
*   **Architectural Trade-off:** As an architect, note the **high memory overhead (200-500MB per instance)**. Running Playwright at scale requires substantial infrastructure compared to heuristic-based tools.

### 3.5 Trafilatura: The Heuristic Surgeon
Trafilatura (specifically the Rust port, `rs-trafilatura`) is the precision tool for scholarly text extraction.
*   **XGBoost Engine:** It utilizes a **181-feature XGBoost model** for page-type classification and a **27-feature model for Confidence Scoring (0.0 to 1.0)**. This allows researchers to automate quality control by flagging low-confidence results for review.
*   **Extraction Profiles:** It provides dedicated extraction profiles for 12 forum platforms and 4 documentation frameworks, ensuring high F1 scores (0.966) across diverse institutional layouts.
*   **Performance:** The Rust implementation offers native speed without a Python runtime, making it ideal for high-throughput transformation layers.

## 4. Python Integration and `uv` Compatibility

From a systems engineering perspective, environment reproducibility is critical.
*   **Dependency Management:** All tools listed provide native Python SDKs compatible with the **`uv` package manager**, ensuring rapid, lockfile-protected installs.
*   **Infrastructure:** While libraries like Crawl4AI and Trafilatura are lightweight, **Firecrawl** requires Docker for self-hosted instances. Playwright requires an additional binary installation step (`playwright install chromium`) which `uv` can orchestrate via post-install scripts.

## 5. Image Extraction & Metadata Discovery

For Job (3), the architecture must handle massive binary data:
*   **Scrapy:** Preferred for large-scale procurement. It manages the entire lifecycle of an image—from URL discovery to S3/GCS bucket storage—with built-in deduplication via SHA1 hashing.
*   **Firecrawl:** Best for capturing visual context. It supports media parsing (PDF/DOCX) and the ability to take full-page or viewport screenshots via its API, allowing researchers to see exactly how an artwork was presented in situ.

## 6. Legal and Ethical Baseline

We adhere to the "Rules of Politeness" derived from the Robots Exclusion Protocol (REP):
*   **Google's robots.txt Protocol:**
    *   **Success (2xx):** Follow rules as provided.
    *   **Server Errors (5xx):** Stop crawling for 12 hours. If the error persists, use cached rules for up to 30 days before assuming no restrictions.
    *   **Size Nuance:** Google enforces a 500 KiB limit for `robots.txt` files; **any content after the first 500 KiB is ignored.**
*   **Ethical Considerations:**
    *   Respect site-wide `disallow` rules for sensitive archival material.
    *   Implement `AutoThrottle` (Scrapy) or rate-limiting to minimize institutional server load.
    *   Validate the Public Domain status of high-resolution assets before storage.

## 7. Final Recommendations and Architecture

### 7.1 Primary Tool Recommendation: Firecrawl
**Firecrawl** is the recommended primary stack. It combines the highest web coverage (96%) with an autonomous research agent and a managed interaction layer. This eliminates the "browser management tax" (managing RAM and Chromium binaries), allowing researchers to focus on data analysis.

### 7.2 Recommended Fallback: Crawl4AI
When data privacy or local cost-efficiency is required, **Crawl4AI** is the fallback. It provides comparable LLM-ready output while remaining within a local Python environment.

### 7.3 Strategic Combinations: The Hybrid Pattern
For large-scale institutional projects, I recommend the **Hybrid Pattern**:
*   **Orchestrator:** Use **Scrapy** to manage the crawl state, request queuing, and binary storage (S3/GCS).
*   **Transformer:** Offload the HTML-to-Markdown transformation to **Firecrawl** or **Trafilatura**. This separates the *crawling* logic from the *extraction* heuristics, ensuring the highest metadata quality.

### 7.4 Minimal Example Architecture
*   **Job 1 (Text):** Firecrawl `/scrape` with `only_main_content=True` for Markdown generation.
*   **Job 2 (Crawl):** Firecrawl `/map` endpoint for rapid URL discovery on JS-heavy museum portals before deep-crawling.
*   **Job 3 (Images):** Scrapy `ImagesPipeline` for high-resolution download with S3 storage and SHA1-based file organization.
