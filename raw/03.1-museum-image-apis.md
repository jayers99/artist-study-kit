# Technical Guide: Automated High-Resolution Artwork Acquisition via Museum APIs and IIIF

### 1. Introduction to Digital Art Retrieval
The programmatic acquisition of high-resolution cultural heritage assets has been transformed by the global "Open Access" movement. Leading cultural institutions now provide public-facing APIs that expose their collections as structured data, shifting the paradigm from manual discovery to automated ingestion. This guide provides a framework for developing Python-based tools that resolve artwork identities, validate licensing schemas, and fetch high-resolution binary images. For the Digital Humanist and Technical Architect, success lies in balancing technical efficiency with strict adherence to the legal and ethical frameworks defined by each institution.

### 2. Understanding IIIF (International Image Interoperability Framework)
The International Image Interoperability Framework (IIIF) is the architectural gold standard for delivering digital heritage. It decouples the descriptive metadata from the pixel-delivery mechanism, allowing for deep-zoom capabilities and sophisticated image manipulation via simple URI patterns.

*   **Presentation API vs. Image API**
    *   **Presentation API (The Manifest):** This API serves a "Manifest" (a JSON object). From an architectural perspective, the Manifest is the assembly instruction set; it contains essential metadata—title, artist, and rights—alongside the structure required for a viewer to display the object.
    *   **Image API:** This API serves the binary pixels. It enables the request of specific regions, sizes, and rotations without requiring the client to download the full-resolution file first.
    *   **Metadata Validation:** A critical endpoint for architects is the `info.json` file (e.g., `/{identifier}/info.json`). This must be queried first to validate image dimensions and supported profiles before initiating a "max" resolution fetch.
*   **The IIIF URL Syntax**
    Standardized requests follow a predictable URI syntax:
    `{scheme}://{server}{/prefix}/{identifier}/{region}/{size}/{rotation}/{quality}.{format}`
*   **Deep Zoom and Transformations**
    *   **Region:** Defines the crop (e.g., `full` or `x,y,w,h`).
    *   **Size:** Using the `max` parameter ensures the highest possible resolution defined by the server. 
    *   **Rotation:** Degrees of clockwise rotation (e.g., `0`, `90`).
    *   **Quality/Format:** Allows for on-the-fly transformations, such as converting a color image to grayscale (`gray`) or specifying the container (e.g., `jpg`, `png`).

### 3. Institutional Source Comparison
The following institutions provide the most robust programmatic access to cultural datasets. Note that several formerly prominent sources are excluded here due to current metadata limitations or lack of verified "Open Access" API documentation in the ground truth.

#### Source-Specific Synthesis
*   **The Metropolitan Museum of Art:** Operates a RESTful service with high-performance search capabilities. It features a strict rate limit of 80 requests per second. No API key is required. Architects should target the `/public/collection/v1/search` and `/public/collection/v1/objects/[objectID]` endpoints.
*   **Art Institute of Chicago:** Features a "unified API" powering all institutional digital touchpoints. It provides IIIF manifests for over 50,000 CC0-designated images.
*   **Rijksmuseum:** Implements the IIIF Image API 3.0 specification via the Micrio platform (`https://iiif.micr.io`). It is highly performant and requires no API key for metadata or binary access.
*   **Harvard Art Museums (HAM):** Access is gated by a unique API key. The API uses a strict concatenation pattern: `URL | Resource | ? | Filter | & | apikey`.
*   **Europeana:** Differentiates between **Personal API keys** (for individual testing and non-production prototyping) and **Project API keys** (for production-level applications operating at scale). Rights are explicitly communicated via the `edm:rights` field.

### 4. Legal and Ethical Baseline
Architectural logic must include automated rights-parsing to ensure compliance with institutional terms.

*   **Public Domain & CC0:** Institutions like AIC and The Met utilize the Creative Commons Zero (CC0) designation, waiving all copyrights to allow unrestricted use.
*   **Rights Statements:** Europeana utilizes "badges" and the `edm:rights` metadata field. If this field is missing, the architect must default to a "restricted" status, assuming standard copyright apply.
*   **Attribution Requirements:** While CC0 does not legally mandate attribution, the Art Institute of Chicago requests the following specific string: *Artist, Title, Date, The Art Institute of Chicago.*
*   **Schema Indicators:** Implementations should parse for `isPublicDomain` (The Met) or the label "CC0 Public Domain Designation" (AIC) as primary triggers for download.

### 5. Strategic Implementation Workflow
A resilient Python tool should follow a three-step resolution and validation strategy:

1.  **Identity Resolution:** Query the search endpoint. To increase precision, utilize boolean filters where available. At The Met, for instance, set `title=true` and `tags=true` in the GET request to filter by specific object attributes rather than a broad keyword search.
2.  **Metadata and License Validation:** Fetch the object record. The tool must parse the JSON response into a strictly typed internal schema:
    *   **Object ID:** `int` (e.g., `objectID: 437133`).
    *   **Rights Status:** `boolean` (e.g., `isPublicDomain: true`).
    *   **Physical Data:** `float` (e.g., `dimensionsParsed` for automated scaling).
3.  **Image Fetching:** Prioritize the IIIF Image API. Use the `info.json` endpoint to confirm the maximum available width/height. Set the URI parameter to `max` for high-resolution acquisition. If IIIF is unavailable, fall back to the institution's direct high-res link, such as `primaryImage` at The Met.

### 6. Recommended Priority Order of Sources
Architects should query sources in the following order to maximize "Open Access" yields and minimize authentication friction:

1.  **The Met:** Highest priority. Massive CC0 catalog (470,000+ records), high-resolution JPEGs, and no API key barrier.
2.  **Rijksmuseum:** High priority. Sophisticated IIIF 3.0 implementation (Micrio) allows for high-res grayscale or color transformations without a key.
3.  **Art Institute of Chicago:** High priority. Clear CC0 designations and extensive IIIF manifest availability.
4.  **Harvard Art Museums / Europeana:** Secondary priority. Significant collections but requires mandatory API key registration, introducing friction for automated "out-of-the-box" tools.
5.  **Wikimedia Commons:** Tertiary fallback. Broad repository but lacks the structured metadata and schema consistency of direct museum APIs.

### 7. Implementation Examples (API/IIIF Requests)

#### The Met (Metadata & Primary Image)
Target the object endpoint to retrieve the `primaryImage` URL:
`GET https://collectionapi.metmuseum.org/public/collection/v1/objects/437133`

#### Rijksmuseum IIIF (Micrio)
Validate dimensions via `info.json` before fetching the binary:
`GET https://iiif.micr.io/RFwqO/info.json`

Request high-resolution assets:
*   **Full-resolution grayscale JPEG:** `GET https://iiif.micr.io/RFwqO/full/max/0/gray.jpg`
*   **Standard high-resolution PNG:** `GET https://iiif.micr.io/RFwqO/full/max/0/default.png`

#### Harvard Art Museums (Concatenation Logic)
Queries must include the key as a trailing parameter:
`GET https://api.harvardartmuseums.org/object?yearmade=2002&apikey=YOUR_UNIQUE_KEY`

### 8. Identified Gaps and Limitations
*   **Copyright Restrictions:** Not all digitized items are open access. At AIC, if an artwork page lacks a download button, the asset must be treated as restricted.
*   **Resolution Variance:** High-resolution is not universal. Implementations must distinguish between `primaryImage` (high-res) and `primaryImageSmall` (web-res) in The Met’s schema.
*   **Institutional Coverage:** Historical European collections are over-represented in Open Access datasets; contemporary and non-Western assets often remain under copyright.
*   **Versioning Risk:** Europeana and other providers do not guarantee that software solutions will continue to function after API upgrades, despite efforts at backwards compatibility. This necessitates robust error handling and version-check logic in any production tool.
